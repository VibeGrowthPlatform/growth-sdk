# Vibe Growth Unity Example

This Unity project validates the Unity SDK against the real local Vibe Growth
backend. It uses the package in `../../..` through Unity Package Manager and
includes a standalone player E2E path that builds and runs a Unity application,
sends representative SDK events, then queries ClickHouse for the persisted
state.

## Requirements

- Unity 2021.3 or newer
- Docker / Docker Compose for the local backend, PostgreSQL, ClickHouse, and Redis
- The SDK E2E config generated by `scripts/validate-sdks.sh --e2e`

## Run The Example Manually

From the repo root:

```bash
bash scripts/validate-sdks.sh --e2e
```

That command starts the backend stack, seeds the SDK E2E app, writes
`/tmp/vibegrowth-sdk-e2e.json`, builds the Unity player app, runs it, and
validates the persisted analytics state.

To open the example app directly:

```bash
open -a Unity --args -projectPath "$(pwd)/examples/unity-player-e2e"
```

Press Play. The example controller reads `/tmp/vibegrowth-sdk-e2e.json` when it
exists, initializes the SDK, and exposes buttons for:

- initialize
- set user id
- track purchase
- track ad revenue
- track first and subsequent sessions
- get remote config

Set `VG_UNITY_EXAMPLE_AUTO_RUN=1` before launching Unity to make the editor
controller send the representative flow automatically after initialization.

## Run Unity Player E2E Only

After the backend stack has been prepared:

```bash
examples/unity-player-e2e/scripts/run_player_e2e.sh
```

Override Unity discovery with:

```bash
UNITY_EXECUTABLE=/Applications/Unity/Hub/Editor/2021.3.45f2/Unity.app/Contents/MacOS/Unity \
  examples/unity-player-e2e/scripts/run_player_e2e.sh
```

## What Gets Verified

The E2E test drives:

1. `VibeGrowthSDK.Initialize`
2. `VibeGrowthSDK.SetUserId`
3. `VibeGrowthSDK.TrackPurchase`
4. `VibeGrowthSDK.TrackAdRevenue`
5. two `VibeGrowthSDK.TrackSessionStart` calls
6. `VibeGrowthSDK.GetConfig`

It then polls ClickHouse for the seeded SDK app until it observes:

- a `devices` row with platform `unity` and SDK version `0.0.1`
- the synced user id on the same device
- purchase and ad revenue rows in `revenue_events`
- first-session and subsequent-session rows in `session_events`
- the backend config response `{}` from `GET /api/sdk/config`
